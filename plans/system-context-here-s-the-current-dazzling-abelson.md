# Rencana: Role Admin/Owner + Database untuk Faedah Shop

## Context

Faedah Shop adalah aplikasi pemesanan & kustomisasi bouquet yang sudah lengkap di sisi pelanggan (katalog 6 produk, kustomisasi, checkout, tracking 4 tahap). Pemilik sekarang ingin menambahkan **role admin/owner** agar bisa mengelola pesanan yang masuk, mengelola katalog produk, dan melihat statistik penjualan.

**Keputusan urutan: Database (Supabase) DULU, baru role admin.** Alasannya:
- Saat ini semua data (`orders[]`, status) hanya disimpan di React state di `src/app/App.tsx` — **hilang setiap refresh**. Panel admin tidak punya data nyata untuk dikelola tanpa persistensi.
- Login admin butuh autentikasi, yang disediakan Supabase Auth secara built-in (lebih aman daripada menaruh password di kode).
- Membangun admin di atas state sementara berarti menulis ulang hampir semuanya saat database masuk.

Database = **Supabase** (Postgres + Auth + Storage). Keputusan yang dikonfirmasi user:
- Auth: **Supabase Auth (email + password)**.
- **Tiga peran terpisah (role-based access)**, setiap akun login sendiri:
  - **Owner** — akses penuh: kelola pesanan, kelola produk, dashboard statistik, **dan kelola akun admin** (buat/hapus admin).
  - **Admin** — terbatas: hanya kelola pesanan & produk. **Tidak** bisa melihat statistik maupun mengelola akun.
  - **Pelanggan** — register/login sendiri. Boleh **lihat katalog & kustomisasi tanpa login**, tapi **wajib login saat checkout** dan untuk melihat **"Pesanan Saya"** (hanya pesanan miliknya sendiri).

> Catatan: Make tidak ditujukan untuk menyimpan data PII sensitif. Data pemesan (nama, telp, alamat) tetap masuk DB untuk fungsi toko, tapi ini perlu disadari user.

## Prasyarat (langkah user)

1. **Connect Supabase** lewat modal `supabase_connect` (akan ditampilkan setelah rencana disetujui). Tanpa ini, implementasi backend tidak bisa dimulai.
2. Setelah terhubung, akan tersedia: `supabase/functions/server/index.tsx` (Hono server), `supabase/functions/server/kv_store.tsx` (autogenerated, jangan diedit), dan `utils/supabase/info.tsx` (projectId + anon key, autogenerated).

## Arsitektur

- **Backend**: Supabase Edge Function (Hono) di `supabase/functions/server/index.tsx`. Semua endpoint diawali prefix yang sudah ada. Data disimpan via `kv_store` (key-value) — pola yang didukung Make — dengan konvensi key:
  - `product:{id}` — data produk
  - `order:{id}` — data pesanan
  - Index ringan untuk listing (mis. simpan array id atau scan prefix lewat helper kv_store).
- **Auth + Role**: Supabase Auth email+password. Setiap user punya **role** (`owner` | `admin` | `customer`) yang disimpan di `kv_store` dengan key `user:{userId}` (berisi role, nama, email) — sumber kebenaran role di server, bukan di metadata yang bisa dimanipulasi client. Pelanggan mendaftar sendiri (signup → role default `customer`). Owner pertama dibuat sekali via seed/endpoint bootstrap; admin dibuat oleh owner. Frontend menyimpan sesi via `@supabase/supabase-js` client. Endpoint terproteksi memverifikasi access token di header `Authorization: Bearer <token>`, lalu **mengecek role** sebelum mengeksekusi (mis. pelanggan ditolak di endpoint admin; admin ditolak di endpoint owner-only).
- **Kepemilikan pesanan**: setiap order menyimpan `userId` pemiliknya. Pelanggan hanya bisa membaca pesanannya sendiri; admin/owner melihat semua.
- **Frontend**: tambah `@supabase/supabase-js`. Aktifkan **react-router** (sudah terpasang v7, belum dipakai) untuk memisahkan area pelanggan dan area admin (`/admin/*`).

## Perubahan Frontend

### Routing (baru)
- `src/app/App.tsx`: bungkus dengan router. Pertahankan flow pelanggan yang ada (catalog → customize → checkout → orders) tapi pindahkan navigasi view-state ke dalam route pelanggan agar tidak merusak UX sekarang.
- Route pelanggan: `/` (katalog — **publik**), kustomisasi (**publik**), `/login` & `/register` (pelanggan), checkout (**wajib login** — bila belum login, arahkan ke `/login` lalu kembali ke checkout), `/pesanan-saya` (**wajib login**, hanya pesanan sendiri).
- Route admin: `/admin/login`, `/admin` (dashboard, owner-only), `/admin/orders`, `/admin/products`, `/admin/accounts` (owner-only).

### Data layer (baru)
- `src/app/lib/supabase.ts` — inisialisasi client dari `utils/supabase/info.tsx`.
- `src/app/lib/api.ts` — helper fetch ke edge function: `listOrders`, `updateOrderStatus`, `createOrder`, `listProducts`, `createProduct`, `updateProduct`, `deleteProduct`, `getStats`. Sertakan token untuk endpoint admin.
- `src/app/lib/auth.ts` — `signUp` (pelanggan), `signIn`, `signOut`, `getSession`, hook `useAuth` yang **mengekspos role user** (owner/admin/customer) sehingga frontend bisa mengatur guard & menu sesuai peran.

### Integrasi alur pelanggan (ubah)
- **Auth pelanggan (baru)**: `src/app/components/CustomerLogin.tsx` & `CustomerRegister.tsx` (form email+password). `Header.tsx` menampilkan status login (nama pelanggan / tombol Masuk) + tombol "Pesanan Saya".
- `src/app/components/Checkout.tsx`: **wajib login** — jika belum login, arahkan ke `/login` lalu kembali. Nama/telp/alamat bisa di-prefill dari profil. Ganti pembuatan order lokal jadi `createOrder` ke backend dengan `userId` pemilik (tetap hasilkan `FLR-xxxxxx`). Upload bukti transfer & gambar referensi ke **Supabase Storage**, simpan URL-nya.
- `src/app/components/Catalog.tsx` & `App.tsx`: produk diambil dari backend (`listProducts`) alih-alih hardcoded — **tetap publik tanpa login**. Seed 6 produk existing ke DB saat pertama jalan (endpoint/seed idempoten).
- `src/app/components/OrderTracker.tsx` → halaman **"Pesanan Saya"**: wajib login, ambil **hanya pesanan milik pelanggan** dari backend, tampilkan tracking status tiap pesanan.

### Komponen Admin (baru) — pakai UI komponen design system yang sudah ada di `src/app/components/ui/`
- `src/app/components/admin/AdminLayout.tsx` — sidebar + guard (redirect ke `/admin/login` bila tak ada sesi). **Menu sidebar dirender sesuai role**: Admin hanya melihat "Pesanan" & "Produk"; Owner melihat semua termasuk "Dashboard" & "Kelola Admin".
- `src/app/components/admin/AdminLogin.tsx` — form email+password (dipakai owner & admin; tujuan redirect setelah login sesuai role).
- `src/app/components/admin/AdminOrders.tsx` — (owner + admin) tabel semua pesanan, filter status, tombol majukan status (Menunggu Konfirmasi → Dalam Proses → Pesanan Siap → Selesai), lihat detail + bukti transfer.
- `src/app/components/admin/AdminProducts.tsx` — (owner + admin) list + form tambah/edit/hapus bouquet (nama, kategori, harga dasar, gambar, sizes, addons).
- `src/app/components/admin/AdminDashboard.tsx` — **(owner saja)** kartu statistik (total pesanan, pendapatan, pesanan per status) + grafik penjualan pakai `recharts`.
- `src/app/components/admin/AdminAccounts.tsx` — **(owner saja)** kelola akun admin: list admin, buat akun admin baru (email + password), hapus akun admin.
- Guard role: route owner-only (`/admin` dashboard, `/admin/accounts`) memeriksa role dari `useAuth`; admin yang mencoba akses diarahkan ke `/admin/orders`.

## Perubahan Backend (`supabase/functions/server/index.tsx`)

Setiap endpoint terproteksi memverifikasi token, lalu mengecek role (owner-only vs admin+owner):
- `GET  /products` (publik) / `POST /products/seed` (idempoten)
- `POST /signup` (publik) — daftar pelanggan baru (role `customer`)
- `GET  /me` (terproteksi) — kembalikan profil + role user yang login
- `POST /orders` (**wajib login pelanggan**, simpan dengan `userId`) / `GET /orders` (pelanggan: hanya pesanannya sendiri) / `GET /orders/:id` (hanya pemilik atau admin/owner)
- `GET  /admin/orders` / `PATCH /admin/orders/:id/status` — **admin + owner**
- `POST /admin/products` / `PATCH /admin/products/:id` / `DELETE /admin/products/:id` — **admin + owner**
- `GET  /admin/stats` (agregasi: total, revenue, breakdown per status) — **owner saja**
- `GET  /admin/accounts` / `POST /admin/accounts` (buat admin via Supabase Admin API) / `DELETE /admin/accounts/:id` — **owner saja**
- `POST /bootstrap/owner` (idempoten, sekali jalan) — buat akun owner pertama bila belum ada

Tipe data di `src/app/components/types.ts` diselaraskan dengan payload backend (kemungkinan tambah field `proofUrl`, `referenceImageUrl`, `createdAt`).

## Styling (design system)
- Semua UI admin baru **wajib pakai variabel CSS dari design system** (`--brand-primary`, `--surface-bg`, `--text-primary`, dll.) dan typography classes (`text-title`, `text-label`, dll.) yang sudah ada di `/styles/global.css` & `theme.css` — sama seperti komponen pelanggan yang ada. Tujuannya agar user bisa mengubah tampilan cukup dengan mengedit CSS.
- **Typography hanya** menggunakan font-face yang sudah didefinisikan di CSS. Tidak ada font baru, dan tidak memakai utilitas Tailwind untuk ukuran/berat/line-height kecuali diminta.
- Gunakan kembali komponen di `src/app/components/ui/` (Button, Badge, InputField, dst.) — jangan bikin elemen UI custom baru bila sudah ada.

## Dependencies
- `@supabase/supabase-js` (install via pnpm)
- `react-router` (sudah ada, tinggal dipakai)
- `recharts` (untuk grafik dashboard; install bila belum ada)

## Tahapan implementasi (urut)
1. Connect Supabase (user) + install deps.
2. Backend: endpoint products (+seed) & orders → uji.
3. Auth pelanggan: signup/login, `/me`, kepemilikan pesanan (`userId`).
4. Frontend pelanggan: katalog publik dari backend; checkout wajib login; "Pesanan Saya" (hanya milik sendiri) → pastikan flow lama tetap jalan.
5. Auth admin/owner + role: bootstrap owner, route `/admin` terproteksi + login + guard berbasis role.
6. Admin orders (lihat semua + ubah status) — admin & owner.
7. Admin products (CRUD) — admin & owner.
8. Owner-only: kelola akun admin (`AdminAccounts`).
9. Owner-only: dashboard statistik (recharts).

## Verifikasi
- **Pelanggan (publik vs login)**: katalog & kustomisasi bisa dibuka tanpa login; menekan checkout saat belum login → diarahkan ke `/login`; setelah register/login → bisa menyelesaikan checkout.
- **Pesanan Saya**: setelah login, buat pesanan → muncul di "Pesanan Saya"; **refresh halaman** → pesanan masih ada (persistensi). Login sebagai pelanggan lain → **tidak** melihat pesanan pelanggan pertama.
- **Auth**: buka `/admin` tanpa login → diarahkan ke login; login owner → masuk dashboard; login admin → masuk halaman pesanan.
- **Role guard**: akun **admin** mencoba buka Dashboard/Kelola Admin → ditolak/diarahkan; menu Dashboard & Kelola Admin **tidak muncul** di sidebar admin. Endpoint owner-only menolak token admin (cek di server, bukan cuma sembunyikan di UI).
- **Admin orders**: pesanan baru muncul di `/admin/orders` (untuk admin & owner); majukan status → pelanggan melihat status terbaru di OrderTracker.
- **Admin products**: tambah produk baru → muncul di katalog pelanggan; edit/hapus berfungsi (admin & owner).
- **Kelola akun (owner)**: owner membuat akun admin baru → admin itu bisa login dan hanya melihat pesanan & produk; owner bisa menghapus akun admin.
- **Dashboard (owner)**: angka statistik & grafik sesuai data pesanan.
- Jalankan lewat preview surface (bukan localhost). Jika perubahan edge function tidak muncul, ingatkan user men-deploy ulang edge function dari halaman Make settings.
